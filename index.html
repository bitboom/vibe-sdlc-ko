Executive Summary

핵심 요약

01

코딩의 인터페이스가 바뀐다

개발자는 점점 문법을 직접 쓰기보다 원하는 결과, 제약, 품질 기준을 설명한다. AI는 이를 실행 가능한 코드로 번역한다.

02

Vibe Coding은 출발점일 뿐이다

빠른 실험에는 유용하지만, 결제·보안·운영 시스템에 그대로 쓰기에는 위험하다. 품질 장치가 붙어야 한다.

03

실무형 해법은 Agentic Engineering

AI를 구현 엔진으로 쓰되, 인간이 아키텍처·테스트·가드레일·리뷰 체계를 설계한다.

04

진짜 역량은 Context Engineering

좋은 프롬프트보다 중요한 것은 모델이 필요한 저장소 맥락, 요구사항, 제약, 테스트 신호를 적시에 제공하는 능력이다.

05

SDLC 전체가 재구성된다

요구사항, 설계, 구현, 테스트, 리뷰, 배포, 유지보수 단계마다 AI 에이전트가 들어오며 단계 간 피드백이 빨라진다.

06

개발자는 지휘자와 오케스트레이터가 된다

직접 모든 코드를 쓰는 역할에서 벗어나, 여러 에이전트의 작업을 분해·위임·검증·통합하는 역할이 커진다.

Main Thesis

문서의 핵심 주장

이 화이트페이퍼는 “AI가 개발자를 대체한다”는 단순한 이야기를 하지 않습니다. 더 정확히는 개발자의 주 인터페이스가 코드 문법에서 의도, 맥락, 판단으로 이동한다고 봅니다.

가장 큰 변화는 새로운 언어, 프레임워크, 클라우드 서비스가 아니라 “코드를 쓰는 일”에서 “의도를 표현하고 지능형 시스템이 그것을 소프트웨어로 바꾸도록 신뢰하는 일”로 옮겨가는 데 있다.

하지만 이 신뢰는 무제한 위임이 아닙니다. 에이전트가 코드를 만들수록 테스트, 리뷰, 관찰성, 보안 가드레일, 컨텍스트 관리가 더 중요해집니다.

이 섹션에서 기억할 것

AI 시대의 핵심 변화는 “코드를 덜 쓴다”가 아니라, 의도·맥락·판단을 더 명확히 설계해야 한다는 점입니다.

Spectrum

Vibe Coding에서 Agentic Engineering으로

Vibe Coding

자연어로 원하는 것을 말하고, AI가 만든 결과를 빠르게 받아 실험하는 방식입니다. 프로토타입과 탐색에는 강하지만, 검증 없이 운영 코드에 넣으면 숨은 부채가 쌓입니다.

Agentic Engineering

AI를 강력한 구현 엔진으로 쓰되, 인간이 요구사항, 아키텍처, 테스트, 제약, 리뷰 루프를 설계합니다. 목적은 빠른 코드 생성이 아니라 반복 가능한 품질입니다.

따라서 이 문서가 권하는 방향은 “바이브 코딩을 금지하라”가 아닙니다. 빠른 탐색은 허용하되, 운영 품질이 필요한 순간부터는 에이전틱 엔지니어링으로 전환하라는 메시지에 가깝습니다.

이 섹션에서 기억할 것

같은 AI 도구를 써도 결과는 다릅니다. 차이는 도구명이 아니라 제약, 테스트, 피드백 루프를 갖춘 사용 방식에서 납니다.

Context Engineering

진짜 실력은 컨텍스트 엔지니어링

원문은 AI 코드 품질이 “프롬프트의 재치”보다 제공되는 컨텍스트의 품질에 더 크게 좌우된다고 봅니다. 컨텍스트 엔지니어링은 코드베이스, 아키텍처, 팀 관례, 제품 의도, 실행 제약을 에이전트가 사용할 수 있는 구조화된 정보로 바꾸는 일입니다.

즉 핵심은 모델을 말재주로 설득하는 것이 아니라, 숙련된 새 팀원이 좋은 일을 하려면 알아야 할 지식을 에이전트가 사용할 수 있는 형태로 제공하는 것입니다. 이 관점이 prompt engineering에서 context engineering으로의 이동을 설명합니다.

핵심 주장: 좋은 질문은 “AI를 어떻게 속여서 좋은 코드를 쓰게 할까?”가 아니라 “새 팀원이 효과적으로 기여하려면 무엇을 알아야 하며, 그 지식을 AI가 쓸 수 있는 형태로 어떻게 인코딩할까?”입니다.

Instructions

에이전트의 역할, 목표, 운영 경계입니다. 무엇을 우선하고 무엇을 금지할지 정합니다. 예: 이 저장소에서는 테스트 없는 기능 추가를 금지한다.

Knowledge

문서, 아키텍처 다이어그램, 도메인 데이터처럼 검색·참조해야 하는 지식입니다. 예: API schema, 제품 정책, 데이터 모델 설명.

Memory

방금 일어난 세션 로그와 프로젝트의 장기 상태를 함께 다루는 기억입니다. 단기 기억은 현재 작업 맥락, 장기 기억은 프로젝트의 안정된 사실을 담습니다.

Examples

few-shot 예시, 기존 코드 패턴, 팀이 선호하는 구현 스타일입니다. 좋은 예시는 자연어 규칙보다 강하게 “이렇게 하라”는 신호를 줍니다.

Tools

에이전트가 호출할 수 있는 API, 스크립트, 외부 서비스와 그 사용 규칙입니다. tool 설명은 단순 목록이 아니라 언제 써야 하는지까지 알려야 합니다.

Guardrails

포맷 규칙, 안전 검증, 절대 어기면 안 되는 하드 제약입니다. secret commit 차단, 권한 범위, 배포 전 테스트 같은 규칙이 여기에 속합니다.

Static Context vs Dynamic Context

Static context는 매번 로드되는 규칙입니다. 예를 들면 system instruction, AGENTS.md, CLAUDE.md, GEMINI.md, 전역 메모리, 페르소나 정의가 여기에 들어갑니다. 항상 적용되지만, 관련 없는 토큰도 매번 비용을 냅니다.

Dynamic context는 필요할 때만 불러오는 정보입니다. task matching으로 로드되는 skill, 실행 중 얻는 tool result, RAG 문서, windowed session history가 여기에 해당합니다. 정보가 필요할 때만 비용을 내므로 더 효율적입니다.

따라서 무엇을 static으로 둘지, 무엇을 dynamic으로 둘지는 단순 설정이 아니라 아키텍처 결정입니다. static이 너무 많으면 토큰을 낭비하고 신호가 희석됩니다. 너무 적으면 agent가 반드시 지켜야 할 규칙을 잊습니다. 좋은 시스템은 이 경계를 코드처럼 리뷰하고 버전 관리합니다.

과도한 static context

모든 정책, 예외, 레퍼런스를 시작 프롬프트에 넣으면 매 호출 비용이 커지고 중요한 신호가 묻힙니다. 오래된 규칙이 계속 남아 context rot도 생깁니다.

잘 설계된 dynamic context

task metadata로 필요한 skill만 열고, 실행 중 필요한 문서와 tool result만 가져옵니다. 에이전트는 가벼운 generalist로 시작해 필요할 때 specialist가 됩니다.

Agent Skills의 의미: 원문은 dynamic context를 관리하는 강력한 패턴으로 Agent Skills를 듭니다. 에이전트는 시작 시 가벼운 메타데이터만 보고, 작업이 맞을 때 전체 지침을 로드하고, 더 깊은 참고 자료는 필요할 때만 가져옵니다. 그래서 하나의 에이전트가 수십 개 전문 능력을 갖되, 실제 사용하는 능력의 토큰 비용만 지불할 수 있습니다.

Agent Skills가 해결하는 문제도 명확합니다. overloaded prompt가 만드는 context rot, LLM에 절차적 기억이 없는 문제, multi-agent 운영 오버헤드, 도구·벤더 간 지식 재사용성 부족을 줄입니다. 그래서 skill은 단순 프롬프트 조각이 아니라 팀의 절차 지식을 패키징하는 단위에 가깝습니다.

이 섹션에서 기억할 것

컨텍스트는 더 많이 넣는 것이 아니라, 필요한 순간에 필요한 정보만 정확히 넣는 설계 문제입니다. Context Engineering은 vibe coding을 agentic engineering으로 바꾸는 다리입니다.

New SDLC

AI가 바꾸는 새로운 SDLC

전통적인 SDLC는 지난 20년 동안 waterfall에서 agile, CI, DevOps, 지속 배포로 이동하며 이미 한 번 압축되었습니다. AI는 이 압축을 더 크게 밀어붙입니다. 구현은 몇 주에서 몇 시간으로 줄어들 수 있지만, 요구사항·아키텍처·검증은 여전히 인간의 판단 속도에 묶여 있습니다.

그래서 원문은 AI 시대의 SDLC를 “기존 프로세스의 더 빠른 버전”이 아니라 단계의 경계가 흐려지고, 반복 주기가 분 단위로 짧아지며, 개발자가 구현자에서 시스템 설계자와 품질 판단자로 이동하는 워크플로로 설명합니다.

변하지 않는 것: 원문은 2026년 중반 기준의 흐름이 12개월 뒤 달라질 수 있다고 말합니다. 다만 인간의 판단, 취향, AI 결과를 검증하는 능력은 계속 남는다고 봅니다.
1

요구사항과 계획

AI는 제품 brief에서 user story를 만들고, 사람이 놓친 edge case를 찾고, 자연어 설명에서 API schema나 prototype을 생성합니다. 요구사항은 팀 간 hand-off 문서가 아니라, 인간과 AI가 대화하며 specification과 초기 구현을 동시에 만드는 과정이 됩니다.

2

설계와 아키텍처

아키텍처는 여전히 가장 인간 중심적인 단계입니다. consistency vs availability, complexity vs flexibility, build vs buy 같은 trade-off는 비즈니스 맥락과 장기 전략을 필요로 합니다. AI는 결정된 아키텍처를 일관된 boilerplate와 module pattern으로 구현하는 데 강합니다.

3

구현

최신 코딩 에이전트는 자연어 설명에서 기능 전체를 만들고, 복잡한 알고리즘과 multi-file change를 수행합니다. 다만 생산성 수치만 보면 안 됩니다. 검증·디버깅·수정 시간이 늘어 AI 사용자가 특정 작업에서 더 오래 걸린 연구도 있습니다. 구현 업무는 “작성”에서 “리뷰·가이드·검증”으로 바뀝니다.

4

테스트와 QA

AI 생성 코드는 최종 산출물뿐 아니라 그 과정도 평가해야 합니다. output evaluation은 컴파일과 테스트 통과를 확인하고, trajectory evaluation은 tool call과 중간 reasoning의 흐름을 봅니다. 테스트와 eval은 AI에게 “정답이 무엇인지” 알려주는 주요 의사소통 수단이 됩니다.

5

코드 리뷰와 배포

AI는 first-pass reviewer로 버그, 스타일 위반, 보안 취약점, 성능 문제를 먼저 잡을 수 있습니다. 하지만 설계·유지보수성·전략적 정합성은 여전히 인간 리뷰가 필요합니다. 배포 파이프라인도 AI-aware해져 위험 예측, health monitoring, 자동 rollback과 연결됩니다.

6

유지보수와 진화

AI는 낯선 legacy codebase를 읽고, 관련 파일을 찾고, 기존 패턴을 존중하며 변경을 적용합니다. “건드리기 너무 위험한 코드”를 refactor, framework migration, deprecated API update, test suite 현대화 대상으로 다시 열 수 있습니다.

Factory Model: 코드를 만드는 시스템 만들기

원문은 이 변화를 factory model로 설명합니다. 현대 개발자의 1차 산출물은 코드 자체가 아니라 코드를 생산하는 시스템입니다. 이 시스템은 specification/context, agents, tests/quality gates, feedback loops, guardrails로 구성됩니다.

공장 관리자가 모든 부품을 손으로 만들지 않듯, 개발자는 assembly line을 설계하고 품질을 관리합니다. 성공 기준은 에이전트에게 단계별 지시를 많이 주는 것이 아니라, 무엇이 성공인지 명확히 정의하고 에이전트가 반복할 수 있게 만드는 데 있습니다.

이 섹션에서 기억할 것

AI가 구현 속도를 높일수록 SDLC의 병목은 요구사항, 아키텍처, 검증, 리뷰 같은 판단 영역으로 이동합니다.

Harness Engineering

모델 주변의 시스템을 설계하라

원문은 AI agent를 모델 그 자체로 보면 잘못된 투자로 이어진다고 경고합니다. 새 모델이 나오면 agent가 좋아지고, 오래된 모델이면 agent가 나빠진다고 설명해버리기 쉽지만, 실제 운영 품질은 그보다 넓은 시스템에 의해 결정됩니다.

모델은 실행 중인 agent의 한 입력일 뿐입니다. prompt, tools, context policy, hooks, sandbox, sub-agents, observability까지 포함한 주변 장치가 바로 harness입니다. 팀이 투자해야 할 표면적도 이 harness입니다.

Agent = Model + Harness. 원시 모델은 agent가 아닙니다. 상태, 도구 실행, 피드백 루프, 강제 가능한 제약을 harness가 제공할 때 비로소 agent가 됩니다.

Instructions & Rule Files

AGENTS.md, CLAUDE.md, GEMINI.md, skill file, sub-agent prompt처럼 agent의 역할과 금지 조건을 정의하는 텍스트입니다.

Tools

agent가 호출할 수 있는 function, MCP server, API와 “언제/어떻게 호출할지”를 알려주는 설명입니다.

Sandboxes

코드가 실제로 실행되는 격리 환경입니다. 무엇에 접근할 수 있고 무엇은 접근할 수 없는지 정합니다.

Orchestration Logic

sub-agent spawning, model routing, specialist hand-off, 어떤 조건에서 어떤 agent가 움직일지 정하는 규칙입니다.

Guardrails / Hooks

tool call 전, file edit 후, commit 전처럼 lifecycle 지점에서 실행되는 결정론적 코드입니다. agent가 자주 잊지만 절대 잊으면 안 되는 것을 막습니다.

Observability

logs, traces, evaluations, cost/latency metering입니다. 관찰성이 없으면 agent가 잘하고 있는지 조용히 drift하는지 알 수 없습니다.

SDLC 안에서 Harness가 하는 일

Configure

요구사항·계획·아키텍처

개발자는 production code가 작성되기 전에 instruction, rule file, architectural constraint, 접근 가능한 API/schema를 설정합니다. 여기서 agent가 깨면 안 되는 기본 규칙이 정해집니다.

Run

구현

모델이 코드를 만들 때 harness는 sandbox, execution environment, tools를 제공해 작업을 안전하고 생산적인 경계 안에 둡니다.

Loop

테스트와 QA

harness는 테스트를 실행할 수 있는 터미널과 환경을 제공하고, 실패 로그를 다시 모델에 돌려보냅니다. 이 구조가 “think → act → observe” 자기수정 루프를 만듭니다.

Observe

리뷰·배포·유지보수

commit 전 hard-coded password를 막는 hook, token cost와 latency 추적, agent drift 관찰, 배포 결정 감사가 harness의 역할이 됩니다.

따라서 vibe coding과 agentic engineering의 차이는 사용하는 도구 이름이 아닙니다. 같은 agent를 쓰더라도 harness를 얼마나 의도적으로 구성하고 적용하느냐가 차이를 만듭니다. agent가 실패했을 때 원문은 먼저 모델을 탓하지 말고 missing tool, vague rule, absent guardrail, noisy context window 같은 구성 실패를 의심하라고 말합니다.

팀의 자산으로서 harness: 재사용 가능한 rule file, skill library, MCP 연결, evaluation harness는 프로젝트를 넘어서 복리처럼 쌓입니다. 잘 만든 harness를 한 번 만들고 여러 번 개선하는 팀이 AI-assisted development에서 더 큰 가치를 얻습니다.
이 섹션에서 기억할 것

agent 품질은 모델만의 함수가 아닙니다. tools, sandbox, hooks, observability까지 포함한 harness가 실무 품질을 만듭니다.

Evaluation

데모가 아니라 평가 기준을 세워라

원문은 vibe coding과 agentic engineering을 가르는 가장 큰 차이로 검증 방식을 듭니다. vibe coding에서는 “실행해보니 그럴듯하다”가 검증이 될 수 있지만, production workflow에서는 충분하지 않습니다.

Agentic Engineering에서는 두 가지 검증이 함께 필요합니다. Tests는 결정론적인 부분을 확인합니다. 같은 입력에 같은 출력이 나오는지, 함수가 계약을 지키는지, 빌드와 타입 검사가 통과하는지를 코드가 판단합니다. 반면 Evaluations는 비결정적인 부분을 봅니다. agent가 적절한 tool을 골랐는지, 올바른 경로로 작업했는지, 최종 응답이 품질 기준을 만족하는지를 rubric, labelled dataset, LM judge로 판단합니다.

Output Evaluation

최종 산출물이 요구사항을 만족하는지 봅니다. 테스트 통과, 빌드 성공, API 계약 준수, 응답 품질, hallucination 여부처럼 결과물 중심의 평가입니다.

Trajectory Evaluation

결과뿐 아니라 과정도 봅니다. agent가 올바른 파일을 읽었는지, 위험한 tool을 쓰지 않았는지, 실패 로그를 보고 수정했는지 같은 tool-use 경로와 compliance를 평가합니다.

Task Success

요구된 작업을 실제로 끝냈는지 평가합니다. “컴파일된다”와 “요구사항을 만족한다”는 다르기 때문에 success criterion을 명시해야 합니다.

Tool Use Quality

필요한 자료를 확인하고 적절한 도구를 썼는지 봅니다. 추측으로 고친 코드가 우연히 통과하는 것을 막는 기준입니다.

Trajectory Compliance

정해진 절차와 권한 경계를 지켰는지 확인합니다. 예: 테스트 전 커밋 금지, secret 접근 금지, 승인 없는 배포 금지.

Response Quality

최종 설명이 사실에 근거하고 재현 가능한지 봅니다. agent가 무엇을 바꿨고 어떻게 검증했는지 추적 가능해야 합니다.

리더를 위한 기준: 작동하는 데모는 agent가 한 번 성공할 수 있음을 보여줄 뿐입니다. eval suite가 통과해야 반복적으로 성공한다고 말할 수 있습니다. 공유 workflow에 넣을 agent는 테스트 커버리지처럼 eval coverage와 명시적 rubric을 가져야 합니다.
이 섹션에서 기억할 것

테스트만으로는 agentic workflow를 충분히 검증할 수 없습니다. 결과와 과정, 두 축의 eval이 있어야 데모가 production discipline으로 바뀝니다.

Developer Role

개발자는 컨덕터와 오케스트레이터가 된다

원문은 AI가 구현 업무를 더 많이 맡을수록 개발자가 두 가지 모드 사이를 오간다고 설명합니다. 하나는 IDE 안에서 AI와 실시간으로 호흡하는 컨덕터, 다른 하나는 여러 에이전트에게 비동기 작업을 맡기고 결과를 검토하는 오케스트레이터입니다.

Conductor: 실시간 지휘

복잡한 로직, 까다로운 디버깅, 낯선 코드베이스처럼 변경 과정을 직접 이해해야 할 때 적합합니다. 개발자는 코드가 만들어지는 과정을 보며 프롬프트와 수정 지시로 세밀하게 방향을 잡습니다.

Orchestrator: 비동기 위임

잘 정의된 버그 수정, 기존 패턴을 따르는 기능 구현, 테스트 생성, 마이그레이션처럼 목표와 검증 기준이 명확한 작업에 적합합니다. 에이전트는 백그라운드나 샌드박스에서 병렬로 일하고, 개발자는 결과를 주기적으로 검토합니다.

오케스트레이터 모드에서는 문법 지식보다 명세 작성, 작업 분해, 결과 평가, 시스템 설계가 더 중요해집니다. 즉 개발자의 가치는 “얼마나 빨리 코드를 치는가”에서 “무엇을 시킬지, 어떤 단위로 나눌지, 무엇을 통과 기준으로 볼지”로 이동합니다.

The 80% Problem

AI 에이전트는 기능의 약 80%를 빠르게 만들어낼 수 있지만, 남은 20%—edge case, error handling, integration point, 미묘한 correctness requirement—는 여전히 깊은 맥락 지식을 요구합니다. 이 마지막 20%가 운영 품질을 결정합니다.

중요한 위험: AI 오류는 더 이상 단순 syntax error에 머물지 않습니다. 비즈니스 로직에 대한 잘못된 가정, 모호한 요구사항에 대한 확인 누락, 빠진 edge case, 장기 유지보수 부담을 만드는 아키텍처 결정처럼 “겉보기에는 맞아 보이는” 개념적 실패가 더 문제입니다.

원문이 권하는 자세는 AI 결과를 모두 받아들이는 것이 아닙니다. AI가 잘하는 명확한 작업의 빠른 구현은 맡기되, 인간의 주의는 모호한 요구사항, 아키텍처 trade-off, correctness 검증에 집중하는 것입니다. 더 빨라지는 방법은 덜 검토하는 것이 아니라, 전문가의 시간을 가장 위험한 부분에 배치하는 것입니다.

이 섹션에서 기억할 것

개발자의 가치는 사라지지 않습니다. 구현보다 specification, decomposition, evaluation, system design 쪽으로 이동합니다.

Economics

AI 개발의 경제학

원문은 AI 개발의 효과를 “개발자가 코드를 얼마나 빨리 쓰는가”로만 보면 부족하다고 말합니다. 엔지니어링 리더에게 더 중요한 지표는 Total Cost of Ownership, TCO입니다. AI 워크플로는 초기 투자 비용인 CapEx와 운영·수정·유지보수 비용인 OpEx의 분포를 바꿉니다.

Vibe Coding: 낮은 CapEx, 높은 OpEx

시작 비용은 거의 없습니다. 구독형 AI 도구와 몇 줄의 프롬프트만 있으면 됩니다. 하지만 구조화되지 않은 파일을 계속 context window에 넣고, 검증되지 않은 결과를 다시 고치게 만드는 prompting loop는 token burn rate를 높이고 first-pass success rate를 낮춥니다.

Agentic Engineering: 높은 CapEx, 낮은 OpEx

초기에는 API schema, deterministic test suite, agent context, guardrail을 설계하는 비용이 듭니다. 대신 에이전트가 governed factory 안에서 동작하므로 산출물은 더 일관되고, 사전 테스트되며, 회사 표준에 맞춰집니다.

Token Burn Rate

모든 입력·출력 토큰은 비용입니다. 관련 없는 대형 context를 매번 넣는 방식은 규모가 커질수록 경제적으로 버티기 어렵습니다.

Maintenance Tax

ad-hoc prompting으로 만든 코드는 구조적 일관성이 약할 수 있습니다. 6개월 뒤 버그가 나면 사람이 AI-generated spaghetti code를 역공학해야 합니다.

Security Remediation

평가 하네스 없이 빠르게 만든 코드는 취약점도 빠르게 만듭니다. production에서 보안 결함을 고치는 비용은 설계 단계에서 잡는 비용보다 훨씬 큽니다.

Model Routing

잘 설계된 factory model은 모든 작업에 frontier model을 쓰지 않습니다. 요구사항·아키텍처 같은 복잡한 작업에는 강한 모델을, 테스트 생성·리뷰·CI 모니터링에는 더 작고 빠른 모델을 배치합니다.

Context Engineering은 재무 전략입니다: 모델에 보내는 정보는 모두 비용입니다. 정확한 AGENTS.md, 아키텍처 가드레일, 동적 skill/tool 호출로 high-signal payload를 만들면 시행착오 루프를 줄이고 운영 토큰 비용을 낮출 수 있습니다.
이 섹션에서 기억할 것

AI 도입의 경제성은 “코드 생성 속도”보다 장기 TCO에 달려 있습니다. 낮은 초기 비용이 높은 운영 부채로 돌아올 수 있습니다.

Where to Start

어디서 시작할까

원문은 AI가 “도입하면 생산성이 자동으로 오르는 기능”이 아니라, 이미 있는 엔지니어링 문화를 증폭한다고 봅니다. 테스트·리뷰·아키텍처 기준이 강한 팀은 더 큰 효과를 얻고, 기준이 약한 팀은 부채도 더 빠르게 쌓입니다.

1

개인 개발자

프로젝트에 AGENTS.md 또는 동등한 규칙 파일을 만듭니다. 처음에는 stack, convention, hard rule, workflow를 10줄 정도로 시작하고, 에이전트가 반복해서 실수한 내용을 규칙으로 추가합니다.

2

작은 workflow부터

반복되는 코드 리뷰, 리서치, 정기 리포트, 테스트 생성처럼 작고 검증 가능한 workflow 하나를 첫 agent 대상으로 고릅니다. 하나를 끝까지 build-evaluate-deploy 해보는 경험이 가장 빠른 학습입니다.

3

테스트와 eval 먼저

코드를 생성하기 전에 테스트와 eval을 작성합니다. 테스트와 eval은 AI에게 의도를 전달하는 계약이며, 자연어 프롬프트보다 더 정확하게 “무엇이 성공인지”를 정의합니다.

4

리더는 eval 기준을 세운다

demo는 한 번 성공했다는 증거일 뿐입니다. 공유 workflow에 들어갈 agent라면 task success, tool use quality, trajectory compliance, hallucination, response quality에 대한 명시적 rubric이 필요합니다.

5

생산 작업과 탐색 작업을 구분한다

vibe coding은 탐색 속도에 맞고, agentic engineering은 production discipline에 맞습니다. 어떤 프로젝트·브랜치·환경에서 어떤 모드를 허용할지 팀 규범으로 분리해야 prototype이 우연히 production이 되는 일을 막을 수 있습니다.

6

조직은 substrate를 먼저 만든다

production agent를 확장하기 전 trajectory/final-response eval, run trace, scoped permission, generated code failure mode에 맞춘 security review를 준비해야 합니다. agent는 이제 단순 도구가 아니라 workflow 참여자입니다.

Structure scales, vibes don't. 구조는 확장되지만 감은 확장되지 않습니다. 탐색에는 바이브 코딩이 유효하지만, 조직이 의존하는 소프트웨어에는 specification, tests, guardrails, human oversight가 선택 사항이 아닙니다.
이 섹션에서 기억할 것

작게 시작하되 production 기준은 초기에 정해야 합니다. 규칙 파일, eval, 리뷰 기준, 권한 경계가 확장의 기반입니다.

Glossary

용어집

원문의 기술 용어는 억지로 번역하지 않고, 한국어 설명과 원문 표현을 함께 둡니다. 실제 팀 문서에서는 영어 용어가 그대로 쓰이는 경우가 많기 때문에 병기하는 편이 오해를 줄입니다.

SDLC Software Development Life Cycle
소프트웨어 개발 생명주기. 요구사항 정의, 설계, 구현, 테스트, 배포, 운영/유지보수까지 소프트웨어가 만들어지고 변화하는 전체 흐름입니다.
Vibe Coding 바이브 코딩
자연어로 의도를 전달하고 AI가 만든 코드를 빠르게 받아 실험하는 방식. 탐색에는 좋지만 운영 품질에는 검증 장치가 필요합니다.
Agentic Engineering 에이전틱 엔지니어링
AI 에이전트를 구현 엔진으로 활용하되, 인간이 아키텍처·테스트·제약·리뷰 루프를 설계하는 실무형 접근입니다.
Context Engineering 컨텍스트 엔지니어링
모델이 올바른 결정을 하도록 필요한 맥락을 선별, 압축, 제공하는 기술입니다. “좋은 프롬프트”보다 “좋은 정보 환경”에 가깝습니다.
Static Context 정적 컨텍스트
매번 로드되는 규칙과 배경 정보입니다. system instruction, AGENTS.md, 전역 메모리처럼 항상 적용되지만 토큰 비용도 항상 발생합니다.
Dynamic Context 동적 컨텍스트
필요할 때만 로드되는 정보입니다. skill, RAG 문서, tool result, windowed history처럼 작업과 관련될 때만 불러와 신호 대비 비용을 높입니다.
Agent Skills 에이전트 스킬
작업이 맞을 때만 전체 지침과 참고 자료를 로드하는 dynamic context 패턴입니다. 하나의 에이전트가 여러 전문 능력을 갖되 필요한 능력의 비용만 지불하게 합니다.
Harness Engineering 하네스 엔지니어링
에이전트가 안전하게 작업하도록 감싸는 instruction, tool, sandbox, hook, guardrail, observability, feedback loop의 설계입니다.
Guardrails 가드레일
에이전트가 넘으면 안 되는 경계입니다. 보안 규칙, 파일 수정 금지 범위, 테스트 통과 조건, 배포 전 검사 같은 하드 제약을 포함합니다.
Hooks 훅
tool call 전, file edit 후, commit 전처럼 특정 lifecycle 지점에서 실행되는 결정론적 코드입니다. 에이전트가 자주 잊지만 절대 놓치면 안 되는 검사를 자동화합니다.
Observability 관찰성
agent run의 log, trace, cost, latency, eval 결과를 추적하는 능력입니다. 관찰성이 없으면 에이전트가 조용히 drift하는지 알기 어렵습니다.
Output Evaluation 출력 평가
최종 산출물이 요구사항을 만족하는지 보는 평가입니다. 컴파일, 테스트 통과, 응답 품질, 결과 정확성 확인이 여기에 해당합니다.
Trajectory Evaluation 과정 평가
최종 결과뿐 아니라 에이전트가 어떤 tool call과 reasoning 경로를 거쳤는지 평가하는 방식입니다. agentic workflow에서는 과정 자체가 품질 신호입니다.
Factory Model 팩토리 모델
개발자의 산출물을 코드 자체가 아니라 코드를 생산하는 시스템으로 보는 관점입니다. specification, agent, test, quality gate, feedback loop, guardrail이 공장의 구성요소가 됩니다.
Conductor 컨덕터
개발자가 IDE 안에서 AI와 실시간으로 상호작용하며 세밀하게 방향을 잡는 작업 모드입니다.
Orchestrator 오케스트레이터
여러 에이전트나 백그라운드 작업에 목표를 위임하고, 결과를 검토·통합하는 작업 모드입니다.
The 80% Problem 80% 문제
AI가 기능의 대부분은 빠르게 만들지만, 마지막 edge case와 correctness 검증에는 인간의 맥락 지식이 필요하다는 문제입니다.
TCO / CapEx / OpEx 총소유비용 / 초기비용 / 운영비용
AI 개발 비용을 초기 구축 비용과 장기 운영·유지보수 비용으로 나누어 보는 경제적 관점입니다.
Model Routing 모델 라우팅
작업 난이도와 비용에 따라 강한 모델과 작은 모델을 적절히 배치하는 운영 전략입니다.
